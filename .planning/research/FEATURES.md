# Feature Research

**Domain:** AI ecosystem monitoring agent with human-in-the-loop review via Slack
**Researched:** 2026-02-24
**Confidence:** MEDIUM — Core features grounded in observed patterns from SignalHub, Crayon, Klue, n8n workflows, and Slack approval patterns. Watchman is a custom internal tool, not a market product, so "table stakes" is defined by what makes the system trustworthy and usable for Lauren, not by competitive positioning.

---

## Feature Landscape

### Table Stakes (Lauren Expects These)

Features that must exist for Watchman to be usable at all. If any of these are missing, Lauren will stop trusting the system or stop using it.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Source registry with configurable scan frequency | Any monitoring tool needs a list of what it monitors and how often — this is the foundation | LOW | Config-file-driven; sources are YAML/JSON entries not database rows |
| Scheduled collectors (RSS, API, scrape) | Without automated pulling, Watchman is just a list of bookmarks | MEDIUM | Cron-based; Python `feedparser` for RSS, `httpx` for APIs, `playwright`/`requests-html` for scraping |
| Signal normalization into structured cards | Raw feed items have inconsistent shapes — Lauren needs a consistent card format to review | MEDIUM | Normalize to: title, source, date, summary, url, raw_score, tier |
| Deduplication across sources | Same launch will appear on Product Hunt, TechCrunch, and an RSS feed — Lauren cannot see the same thing 3 times | MEDIUM | URL hash + fuzzy title matching (Levenshtein or embedding similarity); cluster duplicates before surfacing |
| Slack delivery of signal cards | Lauren lives in Slack; any other review interface will be ignored | LOW | Slack Block Kit with approve/reject/snooze buttons; well-understood pattern |
| Approve / Reject / Snooze actions in Slack | Without these three actions, the review loop has no closure | LOW | Slack interactive components; Block Kit buttons post to webhook; state persisted in SQLite |
| Enrichment triggered on approval only | Enriching every signal wastes LLM tokens on things that get rejected | MEDIUM | Triggered enrichment: capabilities, pricing, API surface, integration hooks |
| Draft tool entry schema on approval | The downstream IcebreakerAI pipeline needs a structured entry — this is the deliverable | MEDIUM | LLM-assisted structured extraction; output validated against IcebreakerAI schema |
| Second Slack approval before pipeline entry | No tool enters IcebreakerAI without Lauren's explicit sign-off on the enriched entry | LOW | Same pattern as step 1; different message format showing enriched card |
| Expandable source registry | The list of sources will grow — adding a new source cannot require code changes | LOW | Config-driven; new source = new YAML entry with type (rss/api/scrape), url, tier, frequency |

### Differentiators (What Makes Watchman Worth Having)

Features that go beyond basic monitoring and make Watchman genuinely useful rather than just another RSS reader with Slack notifications.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Relevance scoring against IcebreakerAI rubric | Generic monitoring tools surface everything; Watchman surfaces only what's relevant to the specific taxonomy and capabilities IcebreakerAI cares about | MEDIUM | Score on: taxonomy fit, novel capability, adoption/traction, credibility, enterprise readiness, integration leverage, cost/benefit; use Claude Haiku for cheap classification |
| Tiered source architecture (Tier 1/2/3) | Treats high-signal structured sources (model releases, official APIs) differently from launch platforms differently from SaaS changelogs — avoids treating a TechCrunch article and a Product Hunt listing identically | LOW | Tier 1: structured/official; Tier 2: launch platforms; Tier 3: SaaS changelogs. Tier affects score weight and frequency |
| Clustering of same-event signals | Without clustering, the same GPT-5 release generates 40 signal cards; with it, Lauren sees one card with N sources attached | HIGH | The hardest part. Options: URL domain deduplication (easy, imperfect), title embedding similarity (better), LLM-assisted clustering (most accurate, most expensive). Start with URL hash + fuzzy title |
| Snooze with expiry | "I've seen this tool — resurface in 30 days if it gains traction" is a real workflow need that plain reject/approve misses | LOW | Store snooze timestamp + expiry in SQLite; cron re-queues after expiry |
| Signal card summary generated by LLM | Raw feed items often have vague titles or SEO-optimized summaries; an LLM-generated 2-sentence summary dramatically reduces Lauren's time-to-decide | LOW | One Haiku call per card; cheap, high value |
| Enrichment depth: API surface + pricing | Capability tools need pricing and API info to be useful to IcebreakerAI — most monitoring tools stop at "here's a link" | MEDIUM | Web scraping + LLM extraction of pricing tiers, API existence, auth method, rate limits |
| Run digest on demand | Beyond scheduled runs, Lauren should be able to trigger "check all sources now" without editing a cron | LOW | Slack slash command `/watchman run` triggers collector immediately |
| Per-source health monitoring | If an RSS feed goes 404 or an API key expires, the system silently stops working — Lauren needs to know | LOW | Track last successful fetch per source; alert if source is stale beyond 2x its frequency |

### Anti-Features (Deliberately Not Building)

Features that seem useful but would undermine Watchman's core value or create maintenance burden disproportionate to return.

| Anti-Feature | Why Requested | Why Problematic | Alternative |
|--------------|---------------|-----------------|-------------|
| Full-auto ingestion without human review | "Why not just automatically add everything scored above 8?" | Lauren's trust in the tool depends on her being in the loop. Auto-ingestion errors are invisible and compound. One bad tool entry in IcebreakerAI is worse than a delayed one | Keep the approve gate permanently. Automate the routing, not the decision |
| Real-time streaming / webhook monitoring | "Wouldn't it be better to get signals the moment they're published?" | Adds infrastructure complexity (persistent connections, webhook registrations, reconnect logic) for marginal benefit. The AI landscape doesn't move at sub-minute granularity | Cron-based collection at appropriate frequency (Tier 1: 4h, Tier 2: 12h, Tier 3: 24h) |
| Web dashboard / admin UI | "It would be nice to see everything in a dashboard" | Slack IS the dashboard. A web UI doubles the interface to maintain, adds auth, and Lauren won't use both | Use Slack home tab if a summary view is ever needed — no separate web app |
| Dev Twitter / X scraping | "Twitter has early AI signal before anywhere else" | Extremely noisy, API access is expensive and unreliable, scraping violates ToS and breaks constantly | Defer. If needed, use a curated list approach via official API with strict relevance filter, not general scraping |
| AI newsletter parsing | "Newsletters like TLDR AI and The Batch have great signal" | Variable HTML structure, email parsing is fragile, deduplication with web sources is hard, newsletters summarize news that's already in RSS | Defer to v2 after core sources are proven. Newsletter authors often post content to their websites too |
| Multi-user review queue | "What if someone other than Lauren wants to review?" | Out of scope for v1. Adds role management, notification routing, and conflict resolution. One reviewer, one queue | Hard-code Lauren's Slack user ID. Generalize reviewer config only if second reviewer is needed |
| Scoring transparency UI | "I want to understand why something got scored 7.3" | Exposing score breakdowns adds complexity to the Slack card and invites arguing with the rubric instead of trusting it | Include the top scoring reason as a single line in the card (e.g., "High: novel capability + enterprise-ready API"). Full score breakdown in logs only |
| Automatic source discovery | "Can it find new sources to monitor?" | The sources list is curated deliberately. Auto-discovery surfaces noisy or low-quality sources and undermines the tiered architecture | Keep source expansion as a human decision. Make adding sources easy (one YAML line), not automatic |

## Feature Dependencies

```
Source Registry
    └──enables──> Scheduled Collectors
                      └──enables──> Signal Normalization
                                        ├──enables──> Deduplication + Clustering
                                        │                 └──enables──> Relevance Scoring
                                        │                                   └──enables──> Slack Card Delivery
                                        │                                                     └──enables──> Approve/Reject/Snooze
                                        │                                                                       └──approve──> Enrichment Pipeline
                                        │                                                                                         └──enables──> Draft Tool Entry
                                        │                                                                                                           └──enables──> Second Approval Gate
                                        │
                                        └──enhances──> LLM Summary Generation (parallel to dedup, not sequential)

Per-Source Health Monitoring ──monitors──> Scheduled Collectors (independent)
Snooze with Expiry ──extends──> Approve/Reject/Snooze (same state machine)
Run on Demand ──triggers──> Scheduled Collectors (bypasses cron, not a separate path)
```

### Dependency Notes

- **Source Registry must exist before any collector runs:** The registry IS the collection config. No registry = nothing to collect.
- **Deduplication must run before scoring:** Scoring a duplicate wastes tokens and inflates Lauren's queue. Deduplicate first, score the canonical signal.
- **Enrichment depends on approval:** Only run the enrichment pipeline for approved signals. This is a cost and quality gate.
- **LLM summary is parallel, not gating:** Summary generation can happen alongside deduplication. It does not block scoring.
- **Second approval gate depends on enrichment:** Cannot present the structured tool entry card until enrichment is complete.
- **Snooze extends the state machine, does not replace it:** Snooze is a third terminal state alongside approve and reject, but with an expiry that re-enqueues.

## MVP Definition

### Launch With (v1)

Minimum viable product to validate that Watchman surfaces useful signals Lauren actually approves.

- [ ] Source registry (config file, 15-20 starter sources across 3 tiers) — without sources, nothing else runs
- [ ] RSS + simple HTTP collectors with cron scheduling — covers Tier 1 and Tier 2 sources
- [ ] Signal normalization into structured cards — required for consistent Slack presentation
- [ ] URL hash deduplication (simple) — prevents Lauren's queue from filling with duplicates immediately
- [ ] Relevance scoring with Haiku (taxonomy fit + novelty) — without scoring, Lauren has to read everything
- [ ] Slack delivery with approve/reject/snooze buttons — the entire review UX
- [ ] Enrichment on approval — produces the draft tool entry
- [ ] Draft tool entry schema output — the actual deliverable
- [ ] Second Slack approval gate — required for quality control before pipeline entry

### Add After Validation (v1.x)

Add once Lauren has used v1 for 2-4 weeks and the core loop is proven.

- [ ] LLM-generated signal card summaries — adds value but costs tokens; validate rubric first, then add polish
- [ ] Fuzzy title clustering (Levenshtein) — upgrade from URL-only dedup once duplicates are confirmed as a problem in practice
- [ ] Per-source health monitoring — important but not launch-blocking; Lauren will notice broken sources
- [ ] Snooze with configurable expiry — `snooze` button can ship in v1 as a simple 30-day default; configurability is v1.x
- [ ] `/watchman run` on-demand slash command — nice to have, cron is sufficient for launch

### Future Consideration (v2+)

Defer until v1 is stable and Lauren's review patterns are understood.

- [ ] Embedding-based clustering — only needed if fuzzy matching produces false positives at scale
- [ ] AI newsletter ingestion — defer until web sources are proven to have gaps
- [ ] Slack home tab summary view — only if Lauren requests a portfolio view of queued vs approved
- [ ] Source tier auto-rebalancing based on approval rate — only with enough data to make it meaningful (3+ months of runs)

## Competitor Feature Analysis

These are not direct competitors (Watchman is a custom internal agent, not a market product) but they reveal what the ecosystem considers standard vs. differentiating.

| Feature | SignalHub | Crayon/Klue | n8n RSS Workflows | Watchman Approach |
|---------|-----------|-------------|-------------------|-------------------|
| Multi-source aggregation | Yes — RSS, web pages | Yes — web crawlers at scale | Yes — via RSS + HTTP nodes | Yes — tiered registry |
| AI-powered noise filtering | Yes — natural language intent | Yes — Klue filters 87% of raw signals | Partial — needs LLM node added | Yes — rubric-based scoring via Haiku |
| Deduplication | Not explicit in docs | Yes — but still produces noise per reviews | URL-based in workflow templates | URL hash + fuzzy title (v1), embeddings (v2) |
| Human review gate | No — delivers to Slack/email directly | Partial — human curation layer optional | No — fully automated delivery | Yes — Slack approve/reject/snooze, non-negotiable |
| Enrichment on approval | No | No — surfaces competitive intel, not tool schemas | No | Yes — capabilities, pricing, API surface |
| Structured schema output | No | No — produces battlecards, not schema | No | Yes — IcebreakerAI-compatible tool entries |
| Downstream pipeline integration | No | No | No | Yes — feeds IcebreakerAI tool registry |
| Configurable per-source frequency | Yes | Not user-configurable | Yes — per-node schedule | Yes — per-source in config file |
| Snooze action | Not observed | Not observed | No | Yes — built into Slack action buttons |

**Key insight from analysis:** No existing product does what Watchman does. Generic monitoring tools (SignalHub, n8n workflows) handle the collection and delivery side but lack the relevance rubric, human gate, enrichment pipeline, and schema output. Competitive intelligence tools (Crayon, Klue) have sophisticated scoring but are oriented toward competitor tracking, not tool registry population. Watchman's differentiator is the combination of opinionated scoring + mandatory human gate + enrichment + structured output — not any one feature alone.
